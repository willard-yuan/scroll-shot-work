# 逆スクロールで長いスクリーンショットを撮りたい——エンジン書き直しを1行で回避した話

iPhoneで長いスクリーンショットを撮りたい人——あるいはアプリ内でスクロールしながらスクリーンショットを撮りたい人——のほとんどは、上から下へスクロールする前提で考えます。ScrollShotの結合エンジンももともとそう設計されました。顛末は<a href="/ja/blog/ios-stitching-engine-v2/">スクリーンショット2枚を繋ぐのはどれくらい難しいのか？何度も作り直したiOS長尺スクリーンショット結合エンジン</a>に書きましたが、要するに 収録 → 等間隔でフレームを抽出 → フレームごとのテンプレートマッチングでオフセットを計算 → 最適な継ぎ目を探す → 切り出して結合、という流れです。このパイプラインは上から下へのケースではすでに堅牢です。

しかし現実はそう簡単にはいきません。ユーザーはチャット——メッセージ、WhatsApp、WeChat——のiPhoneスクロール撮影がしたかったり、スクロールアップして見ていたWebページの途中を保存したかったりします。まさに「アプリ内でiPhone全体のスクリーンショットを撮りたい」という、よく聞かれるシナリオそのものです。そして収録中、彼らの指は上へ滑ります。するとコンテンツは収録の中で逆方向に流れます。ページの一番下から始まり、いわゆる「終わり」が実は古いコンテンツになるのです。これをそのまま上から下用のエンジンに放り込むと、画像は上下逆さまになります。見た目の上が収録の終わり側の古いコンテンツで、下が始めのコンテンツです。

この記事は、ScrollShotがどうやって下から上への結合をサポートしているかについてです。肝は深いアルゴリズムではなく、どれだけ少ないコードで済むかという、極端な設計上のトレードオフにあります。

まずは標準的な上から下への結合デモをご覧ください。これから解説する逆方向の結合は、この仕組みの上に直接成り立っています。

## 最初の直感：別の「上向きエンジン」を作る？

この要件に初めて直面したとき、私の直感はこうでした。下向きの結合エンジンはすでにあるのだから、上向きのものをもう一つ書けばいいのではないか、と。

しかしすぐに、自分を説得して思い止まりました。下向きエンジンの核心は逆方向テンプレートマッチングです。新しいフレーム（img2）の上部からテンプレートを取り、古いフレーム（img1）の下部を探索してオフセットを計算します。これを上向きにひっくり返すと、ほぼすべての座標を反転させなければなりません。テンプレートの取り出し元、探索の走査方向、オフセットの正負の定義、重なり部分（継ぎ目領域）の計算方法……。3,000行のエンジンなら、並行する兄弟実装をもう一つ立ち上げることになり、それぞれに固有のバグと保守の負担が生まれます。さらに悪いことに、下向きエンジンへの修正は上向き側にも反映しなければならず、さもないと両者は静かに分岐していくでしょう。

## 突破口：反転させて、そのまま再利用する

私を止めたのは、当たり前の観察でした。結合エンジンはコンテンツがどう「流れる」かなど気にしません。気にするのは、隣り合う2枚のフレームのうちどちらが上でどちらが下か、ただそれだけです。フレーム列がコンテンツの真の時間軸に沿って並んでさえいれば、同じアルゴリズムがそのまま通用します。下から上への映像は、単に「順番がバラバラ」なだけなのです。

そこでScrollShotの方針はこうです。まずスクロール方向を判定し、それが上向きならフレーム列全体を反転させ、既存の下向きパイプラインに放り込む、と。

```
収録時のフレーム抽出（上向きスクロール、コンテンツは逆方向に流れる）：
  [F1 ページ最下部] → [F2] → [F3] → … → [Fn ページ最上部]
                                 │
             ScrollDirectionDetector が判定：上向き
                                 │
              reversedForStitching() がフレーム順を反転
                                 ▼
  [Fn ページ最上部] → … → [F3] → [F2] → [F1 ページ最下部]
                                 │
              同じ下向き結合エンジンへ（変更ゼロ）
                                 ▼
                   正しい長尺画像（上 → 下）
```

下の図は、この「反転＋再利用」の一連の流れ——フレーム抽出、方向判定、フレーム順反転、下向きエンジンの再利用——を正しい長尺画像に至るまで描いたものです。

![下から上へのスクロール結合アルゴリズム図](/ios_stitching_engine_reverse_scroll_framework_jp.webp)

中核となるコードはたった1行です。

```swift
let normalizedBatch = reverseFrames ? batch.reversedForStitching() : batch
```

`reversedForStitching()` は抽出済みのフレーム参照を反転させ、番号を振り直します。それ以外は一切触りません。反転後の「逆方向に流れていた」コンテンツは再び「順方向」に戻り、下流のテンプレートマッチング、NCC、継ぎ目のロジックは1行たりとも変更する必要がありません。新規の結合コードはゼロです。

> ⚠️ 罠：なぜ「フレーム順を反転させる」のか、そして「エンジン内部に `isReversed` フラグを足す」ではないのか。後者はあらゆる座標系を「順方向／逆方向」という2つのロジックに複製してしまい、バグの表面積を2倍にするからです。変換をデータを放り込む「前」に持ってくることで、エンジンは常に単一の世界に向き合うことになります。

## 方向判定：唯一の新規コード——そしてそれはほぼ「ただ同然」

反転させる以上、前提条件は「ユーザーが上へスクロールした」と認識することです。この手順が重ければ、前述の節約はすべて無駄になります。ScrollShotの方針は軽量で、しかも「ただ同然」と言えるほど賢いやり方です。既存のテンプレートマッチングエンジンを再利用し、パラメータを入れ替えて2回走らせるだけです。

`ScrollDirectionDetector` は収録の中間部分から隣接するフレームを約7ペア一様に抽出します（冒頭の収録カウントダウンや、終わりの停止ボタンは避けます）。そして各ペアに対して、同じマッチャーを2回走らせます。

```swift
let down = directionalScore(previous: prev, current: cur)  // prev→cur は「下へスクロール」か？
let up   = directionalScore(previous: cur, current: prev)  // cur→prev は「下へスクロール」か？
```

鍵となるのは `directionalScore` 内の `guard outcome.offset > 0` という行です。このエンジンの逆方向マッチングは、「新しいフレームの上部にあるテンプレートが、古いフレームのより下の位置に着地したとき」だけ正のオフセットを返します。したがって：

- 実際のスクロールが下向きなら、順方向の呼び出し `down` が成立します（offset > 0、有効）。一方、逆方向の呼び出し `up` は、新しいフレームの上部より下に古いコンテンツを見つけられないため offset ≤ 0 となり、無効としてスコア0になります。
- 実際のスクロールが上向きなら、逆になります。`down` が無効で、`up` が有効です。

方向の判定基準は、実は「2つのスコアが対称か」ではなく「**どちらの呼び出しが有効か**」です。初稿では私はうっかり「完全に対称である」と書いてしまい、同僚に指を差されて「有効と無効がどうして対称になり得るのか」と問われました。この記事で最も正したかった誤解が、そこにあります。

各ペアの `down` / `up` スコアを得たあと、ペアごとのスコア算出自体は次のように動きます。

```
score = probability×0.55 + nccBoost×0.20 + overlapRatio×0.15 + consensusBoost×0.10
score ×= sadPenalty          // 重なりがぼやけるほどスコアを割引く
有効とみなすスコアは 0.26 以上でなければならない
```

次に投票です。1ペアの中で、`up ≥ down × 1.25`（あるいは絶対値で0.08以上上回る）なら上向きに1票、さもなければ下向きに1票を投じます。最終的な判定は：

- `upwardVotes ≥ downwardVotes + 2`、かつ
- `upwardScore ≥ downwardScore × 1.25`

この2条件を満たして初めて上向きスクロールとみなされます。この一連の処理は、320pxに制限した低解像度のサムネイル上でバックグラウンドスレッドに走り、総処理時間への影響はほぼ無感覚です。

## 迷ったら、ユーザーに委ねる

判定は100%信頼できるわけではありません。2〜3ピクセルしか動かなかったほぼ静止した収録では、方向のシグナルが非常に弱くなります。何も考えずに反転させるのは、反転させないより悪いのです。そこで仕切り（ゲート）を用意しました。`shouldAskForReverseConfirmation` は、確信度 ≥ 0.72 かつ有効なサンプルが3ペア以上のときだけ発動し、軽量な確認を出します。「上へスクロールして収録したと検出しました——結合を反転しますか？」。ユーザーが反転をタップすれば `reverseFrames: true`、迷ったら元の順序のまま結合します。エンジンが方向を誤って黙って間違った画像を作ることはありません。

## 2つのモードを並べて見る

| | 上から下 | 下から上 |
|---|---|---|
| 収録のジェスチャ | 指を下へ滑らせる | 指を上へ滑らせる |
| フレーム抽出順 | 自然な時間軸 | 自然な時間軸（コンテンツは逆方向に流れる） |
| 方向判定 | 不要 | 収録中盤から7ペア抽出、マッチャーを2回走らせる |
| 結合エンジン | 既存のまま | 既存のまま（フレーム順反転後に再利用） |
| 新規コード | — | 方向判定 ＋ 1行の反転のみ |
| フォールバック | — | 確信度が低いとき確認を促す |

## まとめ

振り返ると、下から上への結合はScrollShotの中に「固有のコード」をほとんど持っていません。下向きエンジンに完全に寄生しているのです。「まず順序を正規化してから、同じエンジンを再利用する」という考え方は、後に新しい結合モードを追加するときの既定の出発点になりました。次に横向きやジグザグの結合をする際も、ひとから作るのではなく、まず問題を「1次元の下向き」へとねじ戻す方法を考えるはずです。

## FAQ：iPhoneアプリでスクロール撮影するには

**ScrollShotはメッセージやWhatsAppのようなiPhoneアプリでスクロール撮影できますか？**
はい。WebページのiPhone長尺スクリーンショットでも、メッセージやWhatsAppのスレッドのiPhoneスクロール撮影でも、ScrollShotが画面を収録しフレームを自動で結合します。システムの「全体」ボタンに頼るのではなく画面収録でキャプチャするため、どのアプリ内でも動作します。まさに「アプリ内でiPhone全体のスクリーンショットを撮りたい」という、よく聞かれるケースそのものです。

**チャットを収録中に上へスクロールしたらどうなりますか？**
それこそがこの記事の主題です。メッセージユーザーが好むiPhoneスクロール撮影も、WhatsAppファンが上へスクロールして撮るスクロール撮影も、どちらもコンテンツが逆方向に流れることを意味します。ScrollShotは上向きのスクロール方向を検出し、フレーム順を反転させ、同じ下向き結合エンジンを再利用します。そうして私たちが何も書き直さずとも、正しいiPhoneスクロール撮影が得られるのです。

**標準のiOSスクリーンショットとどう違いますか？**
標準のスクリーンショットでは、アプリ内の全体ページを必ずしもキャプチャできるとは限りません。ScrollShotはiPhone長尺スクリーンショットとスクロール撮影のシナリオのために作られています。一度収録し、好きな方向へ（上でも下でも）スクロールするだけで、1枚のシームレスな長尺画像が得られます。
